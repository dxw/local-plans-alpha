Publish your development management plans

Section 19(1B) - (1E) of the Planning and Compulsory Purchase Act 2004 sets out that each local planning authority must identify their strategic priorities and have policies to address these in their development plan documents (taken as a whole).

A Local Development Scheme is required under section 15 of the Planning and Compulsory Purchase Act 2004 (as amended). This must specify the development plan documents (i.e. local plans) which, when prepared, will comprise part of the development plan for the area. The Local Development Scheme must be made available publicly and kept up-to-date.

Local Development Schemes must also be produced in compliance with any data standard for this purpose published by MHCLG. Up-to-date and accessible reporting on the Local Development Scheme in an Authority’s Monitoring Report is an important way in which authorities can keep communities informed of plan making activity.

This guidance will help local planning authorities ensure that the data in their Local Development Schemes and development plan documents are:

  • published in the agreed format and contain the required data
  • kept up to date
  • easily found online

The data will give policy makers better insights into how local plans are supporting new development and local infrastructure. We have designed the data format to cause as little disruption as possible to existing processes.

Please share any feedback on this guidance by emailing DigitalLand@communities.gov.uk

Make your data findable, usable and trustworthy

A local plan sets out all local planning policies and decisions for authorities. It identifies how land is used, decides what will be built where and is used to determine all local planning decisions. Every local planning authority is required to update its local plan every 5 years.

But for data to be useful it must be easy to find, use, understand and trust. Local planning authorities should ensure that their developer contributions data is all of these by following this specification for how to format, label and publish it. The following guidance gives a high-level overview of the data format and the publication process.

As a local planning authority, you must review your development management data at least once a year. However, you should update your register as soon as there is any change to your local development scheme or plans to increase trust in the data.

Publishing developer contributions: 3 steps

Planning authorities must publish their development plan data once a year but are encouraged to do so as frequently as they can so that datasets are up to date.

This guidance applies to the publication of development plan data going forward – you are not required to republish historical data so that it complies with this guidance.

Publishing is a 3 step process:

  1. Create 2 development management CSV files to store the data
  2. Upload each of these files to your organisation’s website and give them a persistent URL, then clearly link to them from your local plan web page
  3. Submit the URLs you created in Step 2 to the national register of developer contributions

Step 1: Create your CSV files

To complete step 1 you must be able to create or amend CSV files, for example by using spreadsheet software.

Development management data must be entered in 2 separate CSV files. A CSV file (or comma separated value file) is a universally recognised file format for storing tabular data in plain text. Storing the data in 2 separate files rather than 1 file reduces duplication and makes the data easier to use and maintain.

We’ve created a CSV template for each of the 2 files:

  • CSV 1: local development scheme
  • CSV 2: policies

If it helps, you can use the above example files and enter your data. You must follow the guidelines below, then ‘save as .csv file’. You can use software such as Microsoft Excel, Google Sheets or Apple Numbers, as long as they meet the requirements of this guidance.

Each of the CSV files must:

  • be named using the convention specified in each section below
  • contain certain column headers (written exactly as shown, in lowercase)
  • only contain entries that conform to the constraints described below

Find out more about creating a CSV file.

CSV file 1: Development Plan Documents

A development plan document is any document that contains policies that address strategic priorities for development under the Planning and Compulsory Purchase Act 2004.

The Local Development Scheme sets out the development plan documents which, when prepared, will comprise part of the development plan for the area. It sets out the expected dates that each development plan document will enter the specific phases in its progress to becoming adopted.

The development plan documents CSV file records, as seperate lines, each status that a development plan document will move through on its progress to becoming adopted.

Do not delete or overwrite old or superseded entries, but simply fill in the ‘end-date’ field.

Name your CSV file

Development Plan Documents and their expected dates must be listed in a CSV file named exactly as follows, but with the actual date you created the file instead of YYYYMMDD:

development_management_document_YYYYMMDD.csv

Column headers:

dpd-id

Create a unique identifier for the document. By ‘unique’ this means it should not be used for anything else in your organisation (for example ‘dpd1’). The unique identifier will be used to map against the policies contained within the dpd.

organisations

Find your organisation in this list (in most cases this will be a local planning authority). Text must follow the same letter casing, with no spaces. Norfolk’s local planning authority, for example, would be:

local-authority-eng:NFK

If a DPD relates to multiple organisations, for example a joint local plan, enter each organisation separated by a space.

Read the MHCLG Digital Land team’s data principles for more information on dates.

status

Each development plan document should be recorded on multiple lines to reflect the stages that the document will go through on its progress to adoption. Each line relates to the status that the document will move through. For new development plan documents record the status on separate lines:

  • ‘regulation18’
  • ‘regulation19’
  • ‘submission’
  • ‘adopted’

For existing development plan documents that are withdrawn, you can add a new line with the status ‘withdrawn’

entry-date

Enter the date that you record this data (for example today’s date), in the format YYYY-MM-DD. For 1 February 2019, for example, you should enter 2019-02-01.

start-date

Enter the date that the development plan document is expected to enter the status relating to this line, or the date on which this has already happened.

If this date changes, then a new line should be entered with the new date in this field. Do not delete or overwrite old or superseded entries.

Enter the date in the same format as the entry-date column.

end-date

Leave this blank if this status has not yet completed. If it has, enter the last day this status was in effect, in the same format as the entry-date column.

dpd-name

Enter the name of the development plan document

document-url

Enter the web address that links directly to the actual development management document. If the document is not published, leave this blank.

category

Enter the strategic category or categories from the National Planning Policy Framework that are addressed in the development plan document from the list below. If there are multiple categories, separate each category with a space.

  • ‘HO’ housing
  • ‘EM’ employment
  • ‘RE’ retail
  • ‘LE’ leisure
  • ‘OC’ other commercial development
  • ‘TI’ transport infrastructure
  • ‘TE’ telecommunications
  • ‘SE’ security
  • ‘TI’ transport infrastructure
  • ‘WM’ waste management
  • ‘WW’ wastewater
  • ‘FR’ flood risk
  • ‘WS’ water supply
  • ‘CC’ coastal change management
  • ‘ME’ provision of minerals and energy
  • ‘CF’ community facilities
  • ‘CN’ conservation and enhancement of the natural environment
  • ‘CB’ conservation and enhancement of the built environment
  • ‘CH’ conservation and enhancement of the historic environment/li>

shape

Enter the polygon data that relates to the geographic area covered by the development plan document in GeoJSON format

description

Enter a short (<100 character) description of the development plan document

policy-period-start

Enter the date that the policies in the development plan document will come into effect.

Enter the date in the same format as the entry-date column.

policy-period-end

Enter the date that the policies in the development plan document will cease to be in effect.

Enter the date in the same format as the entry-date column.

update-reason

Enter a short reason for updating the development plan document or its status. If this is the first entry for the status, you can leave this blank

evidence-base-url

Enter the web address that links directly to the evidence base that supports this development management document. If the evidence base is not published, leave this blank.

type

Enter the type of development plan document

  • ‘AAP’ Area Action Plan
  • ‘CS’ Core Strategy
  • ‘DM’ Development Management
  • ‘LP’ Local Plan
  • ‘M&W’ Minerals & Waste
  • ‘Other’ all other types
  • ‘PR’ Partial Review
  • ‘SA’ Site Allocations

CSV file 2: Policies

Policies are contained within development plan documents and they set out the specific plans and measures to meet a local planning authority’s priorities.

Do not delete or overwrite old or superseded policies, but simply fill in the ‘end-date’ field<.

Name your CSV file

Policies must be listed exactly as follows, but with the actual date you created the file instead of YYYYMMDD:

policy_YYYYMMDD.csv

Column headers:

policy-id

Create a unique identifier for the policy. By ‘unique’ this means it should not be used for anything else in your organisation (for example, policy-1).

dpd-id

Enter the unique identifier for the development plan document that this policy is contained within.

policy-name

Enter the name of the policy

document-url

Enter the web address that links directly to the policy. If the policy does not have it’s own web address, leave this blank.

strategic

Enter either ‘strategic’ or ‘non-strategic’ to record whether policy is strategic.

category

Enter the strategic category or categories from the National Planning Policy Framework that are addressed in the policy from the list below. If there are multiple categories, separate each category with a space.

  • ‘HO’ housing
  • ‘EM’ employment
  • ‘RE’ retail
  • ‘LE’ leisure
  • ‘OC’ other commercial development
  • ‘TI’ transport infrastructure
  • ‘TE’ telecommunications
  • ‘SE’ security
  • ‘WM’ waste management
  • ‘WS’ water supply
  • ‘WW’ wastewater
  • ‘FR’ flood risk
  • ‘CC’ coastal change management
  • ‘ME’ provision of minerals and energy
  • ‘CF’ community facilities
  • ‘CN’ conservation and enhancement of the natural environment
  • ‘CB’ conservation and enhancement of the built environment
  • ‘CH’ conservation and enhancement of the historic environment

related-policy-id

Enter the unique id of any related policies that should be read in conjunction with this policy. Separate multiple policies with a space.

shape

Enter the polygon data that relates to the geographic area covered by the policy in GeoJSON format

unit-type

If the policy specifies a specific number or target, for example housing target numbers, enter the type of unit, as listed on our plan unit type page.

More unit types will gradually be added to the plan-unit-type.csv file, which MHCLG will maintain for your reference.

unit

If you have entered a unit-type, then enter the numeric value over the plan period

entry-date

Enter the date that you record this data (for example today’s date), in the format YYYY-MM-DD. For 1 February 2019, for example, you should enter 2019-02-01.

start-date

Enter the date that the policy comes into (or came into) effect.

Enter the date in the same format as the entry-date column.

end-date

Leave this blank if this policy is still in effect. If the policy is changed or is no longer in effect, enter the date that this policy is no longer in effect.

Enter the date in the same format as the entry-date column.

description

Enter a short (<100 character) description of the development plan document

evidence-base-url

Enter the web address that links directly to the evidence base that supports this policy. If the evidence base is not published, leave this blank.

Step 2: Update your local plan web page

To complete step 2 you must be able to upload the files created in step 1 to your local planning authority’s website and edit (or create) your local plan web page. If you are not able or authorised to do this, speak to someone who is (this might be the person who updates your local planning authority’s brownfield site register or planning application web pages).

Upload each CSV file to your local planning authority’s website. Make sure the URL for each CSV file is persistent (a web address that will not change over time). As an example, Norfolk’s local planning authority might use the following persistent addresses (note that these are examples, not live data):

https://www.norfolk.gov.uk/development-management-document

https://www.norfolk.gov.uk/policy

When you upload the files you should keep any older files online – it’s important to retain a history of all development management documents and policies

You should then update your local planning authority’s local plan web page to make the CSV files publicly available.

The page must list the persistent URLs as defined above. The URLs must be written out in full. (The URLs should not change. However, if they do, you must tell us.)

Licensing

You must state on your web page that the data is provided under the Open Government Licence.

Step 3: Submit the URLs you created in Step 2 to the national register of developer contributions

MHCLG will maintain a national register of development plan documents and policies.

Email the persistent URL for each CSV file to DigitalLand@communities.gov.uk and we will add them to the national register. You must tell us if these URLs ever change.
